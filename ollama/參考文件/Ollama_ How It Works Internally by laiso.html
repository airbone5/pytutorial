Ollama and llama.cpp

https://github.com/ollama/ollama

Ollama internally uses llama.cpp to execute inference for LLM text generation. By using quantized GGML/GGUF models for llama.cpp, it enables running models that normally require high-performance GPUs and RAM on regular user PCs.

Client and Server

Ollama consists of two main components: the client and the server. The client is the frontend that the user interacts with, such as a CLI (command line interface). The server is a backend service implemented in Go that provides a REST API. This is called the ollama server, and the client communicates with this backend service.

Ollama’s Internal Functions

Ollama uses CGoto call functions from the llama.cpp library. For example, the ollama create command used to create custom models calls the llama_model_quantize() function from llama.cpp to execute model quantization.

https://github.com/ollama/ollama/blob/105186aa179c7ccbac03d6719ab1c58ab87d6477/llm/llm.go#L23-L39

Additionally, Ollama internally runs ollama_llama_server which has the web server functionality of llama.cpp. This allows text generation to be executed on the llama server side implemented in C++, and the ollama server receives and post-processes the results.

https://github.com/ollama/ollama/blob/105186aa179c7ccbac03d6719ab1c58ab87d6477/llm/server.go#L264-L291

Device Drivers and GPU Support

Ollama supports GPU acceleration using Nvidia and AMD GPUs. On Apple devices, it supports GPU acceleration via the Metal API. It automatically selects the optimal model based on the execution environment.

Model Files

Ollama has model files that can be used to create and share models. These contain information such as the base model, parameters, prompt templates, system messages, LoRA adapters, licenses, message history, etc. The ollama pull command downloads models from Ollama's registry. By default, binary files hosted at registry.ollama.ai are downloaded and saved under ~/.ollama.

Security Note

In the past, there was a vulnerability (CVE-2024–37032) that allowed remote code execution via malicious HTTP requests to the Ollama server. This vulnerability could potentially be exploited remotely in Ollama Docker deployments where the API server was exposed externally.
